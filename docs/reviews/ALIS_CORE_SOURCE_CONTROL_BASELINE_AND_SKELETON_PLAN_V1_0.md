# ALIS Core Source-Control Baseline and Target Skeleton Plan v1.0

## 1. Purpose

This document records a planning-only source-control baseline and target
skeleton plan for ALIS Core repository migration.

It does not change the Git baseline. It does not stage, commit, move, rename,
delete, refactor, clean, or restructure any file. It does not create target
skeleton folders or implement engines.

The purpose is to identify the current repository state that must be resolved
before migration and to describe the target skeleton that may be created later
after independent review.

## 6. Generated Artifact Concerns

Generated artifact concerns are material blockers for migration:

- Python bytecode is present as untracked `.pyc` files under backend and tests.
- Runtime logs exist under `data/raw/*/logs/` and `data/validated/`.
- Generated reports exist under `reports/generated/`.
- Production-candidate output exists under `data/publishing/`.
- Root `evidence_package_output.json` is generated output outside an artifact folder.
- Some generated data may also serve as historical audit evidence, so cleanup
  must not happen without retention review.

Generated artifacts should be classified into one of:

- commit as approved evidence fixture
- move later to an approved artifact store
- ignore locally
- archive outside source control
- delete later with explicit approval

No generated artifact is cleaned by this milestone.

## 14. Risks

| Risk | Severity | Evidence | Mitigation |
| --- | --- | --- | --- |
| Migration begins from dirty source-control state | Critical | 13 modified tracked files and 250 untracked paths before this document. | Establish reviewed commit/ignore/delete decisions first. |
| Generated artifacts are committed as source | High | `.pyc`, logs, reports, raw data, and production candidates are present. | Approve artifact policy before commit or cleanup. |
| Source candidates remain untracked during migration | High | Many backend and test source files are untracked. | Commit accepted source/test candidates before movement. |
| Skeleton folders imply premature architecture | Medium-high | Target `src/alis_core` and `src/godatabank` do not exist yet. | Create skeleton only after independent review. |
| Line-ending churn pollutes baseline | Medium | Git reported LF-to-CRLF warnings for seven tracked files. | Review `.gitattributes` and line-ending policy later. |
| Ambiguous files get committed without ownership | Medium-high | Pipelines, validators, publishing, templates, and source registry need review. | Require human review before commit. |
| Deleting generated files loses audit evidence | Medium-high | Some generated outputs may be evidence for prior reviews. | Do not delete without retention decision. |
| Test evidence becomes unreliable | Medium | Tests and pycache are mixed under untracked `tests/`. | Commit test sources, exclude caches, then map tests by owner. |

## 15. Abort Conditions

Abort any future skeleton or migration milestone if:

- modified tracked files remain unreviewed
- untracked source candidates remain uncommitted or undispositioned
- generated artifacts do not have an approved commit/ignore/delete policy
- `.pyc` or `__pycache__` cleanup is requested without explicit approval
- line-ending normalization would occur implicitly during commit
- target skeleton creation is combined with code movement
- target skeleton creation is combined with import changes
- engine package creation is used to begin implementation
- GoDataBank product folders are created before product/core boundary approval
- `artifacts/` or `legacy/` are created before retention and legacy policies are approved
- any command would stage, commit, reset, delete, move, rename, or restructure files without explicit authorization
