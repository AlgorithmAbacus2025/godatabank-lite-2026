# ALIS Core Repository Engineering Audit v1.0

## Executive Summary

This audit reviews the current GoDataBank repository as an independent engineering assessment against the frozen high-level architecture:

```text
Ambanica Technologies Ltd
  -> ALIS Core
    -> Core Engines
      -> Abacus
      -> Arya
      -> Vyasa
      -> Hermes
      -> Thoth
      -> Argus
      -> Apollo
      -> Aegis
  -> GoDataBank
```

The repository contains a substantial working evidence, validation, visualisation, publishing, and local production-candidate chain. The strongest areas are schema-driven evidence modelling, mocked unit tests, static publishing safety checks, and milestone documentation. The weakest areas are architectural separation, repository hygiene, explicit engine boundaries, and alignment between frozen architecture language and current code layout.

The current repository is best described as a working GoDataBank prototype where ALIS Core implementation and GoDataBank product concerns are mixed together. It is not yet organised as a clean ALIS Core platform with GoDataBank as a separate application built on top.

The audit did not modify code, tests, data, generated output, deployment configuration, or project structure. The only file created by this audit is this report.

The project has clear accumulated capability, but the current folder layout does not yet express the frozen ALIS Core architecture. The strongest structural unit is `backend/alis/`, but that package currently contains retrieval, evidence packaging, statistical validation, visualisation, static publishing, Cloudflare upload inspection, production-candidate generation, and evidence capture. ### Logical Separation

Logical separation is partial.

Evidence of separation:

- `backend/alis/connectors/` contains World Bank and ONS connector code.
- `backend/alis/schemas/` contains evidence and visualisation Pydantic models.
- `backend/alis/validators/` contains statistical comparison validation.
- `backend/alis/visualisation/` contains chart-spec, SVG, and HTML embedding layers.
- `backend/alis/publishing/` contains static publishing, manifest, site assembly, smoke-test, and production-candidate modules.
- `backend/abacus/` contains deterministic classification code.
- `backend/summary/` and `backend/publisher/` contain earlier summary and HTML report generation layers.

Evidence of boundary mixing:

- Publishing and visualisation code is located under `backend/alis/`, even though ALIS contract documentation states ALIS should not chart or publish.
- GoDataBank-specific production candidate output code is under `backend/alis/publishing/`.
- Product data, generated reports, production candidate output, and evidence files are present in the same repository tree as reusable backend code.
- There is no explicit `alis_core/` package and no explicit `godatabank/` application package.

### Engine Mapping Conclusion

Only Abacus exists by name as an engine-like module. The remaining proposed engines are not implemented explicitly and do not have repository-defined contracts.

## GoDataBank Separation

GoDataBank is currently closer to option B:

```text
B. GoDataBank is mixed together with ALIS implementation.
```

Evidence:

- `backend/alis/publishing/production_candidate_static_output.py` and related candidate-output modules contain product/static-site concerns under the ALIS package.
- `backend/alis/visualisation/` contains charting concerns under ALIS.
- `backend/alis/publishing/` contains report page, index, manifest, site assembly, smoke-test, and Cloudflare upload inspection concerns.
- GoDataBank generated reports and candidate output are present under `reports/` and `data/publishing/` in the same repository tree.
- There is no separate `godatabank/` product application layer.
- There is no separate `alis_core/` platform package.

The current arrangement has enabled rapid milestone progress, but it creates architectural ambiguity. A future clean separation should not begin with feature work. It should begin with a documented boundary register that maps existing modules to ALIS Core engines and GoDataBank product responsibilities.

## Testing Assessment

### Evidence

Static test inventory found 197 pytest test functions under `tests/alis/`.

The latest milestone review evidence records:

```powershell
python -m pytest -p no:cacheprovider tests/alis/
```

Result recorded in project review documents:

```text
197 passed
```

This audit did not re-run pytest because the user requested review-only work and allowed no project file changes except this audit report. Even with `-p no:cacheprovider`, Python test execution can create or update bytecode caches in normal environments. Existing recorded pytest evidence was reviewed instead.

### Test Strengths

- Broad unit test coverage exists for schemas, connectors, statistical gatekeeping, mapping, chart specification export, SVG rendering, HTML embedding, static report fragments/pages/indexes, manifest validation, static site assembly, smoke testing, production candidate output, Cloudflare upload boundary inspection, and evidence capture.
- Many tests use monkeypatching and avoid external API calls.
- Safety behaviours are repeatedly tested: blocked files, unsafe HTML, unsafe SVG, invalid manifests, invalid chart specs, and invalid integrity states.
- The test suite appears to have grown consistently with each frozen milestone.

### Test Gaps

- No coverage report was found.
- No CI configuration was found.
- Tests are grouped under `tests/alis/` even when testing publishing, visualisation, deployment-boundary, and production-candidate modules.
- At least some connector testing appears to exercise live connector behaviour, which can reduce determinism if not isolated or marked.
- There is no explicit test matrix for Python versions, operating systems, or network-isolated execution.
- There are no explicit package/import boundary tests for the proposed ALIS Core engine architecture.

## Recommended Next Milestone

Recommended next milestone:

```text
ALIS Core Boundary Register v1.0
```

Objective:

Create a documentation-only boundary register that maps every existing backend module, schema, validator, pipeline, visualisation module, publishing module, generated artefact, and test group to one of:

- ALIS Core shared platform
- Abacus
- Arya
- Vyasa
- Hermes
- Thoth
- Argus
- Apollo
- Aegis
- GoDataBank product layer
- generated artefact
- obsolete candidate
- temporary artefact

Rationale:

- It follows the frozen architecture without expanding features.
- It avoids premature refactoring.
- It creates the evidence needed to separate GoDataBank from ALIS Core safely.
- It resolves ambiguity before any code movement or cleanup.
- It gives the next engineering action a traceable basis.

Do not begin feature expansion, deployment, new connectors, AI work, dashboards, or architecture redesign until this boundary register exists and is reviewed.

## Audit Conclusion

The repository demonstrates serious engineering effort and a meaningful local evidence-to-static-output chain. However, it is not yet a clean ALIS Core repository with GoDataBank separated as a product. The immediate engineering priority is not more functionality. The safest next step is to freeze a precise boundary register, then use that register to guide future repository hygiene and separation work.
